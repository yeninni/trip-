# Git Workflow

이 저장소는 앞으로 아래 규칙으로 운영합니다.

## 브랜치 역할

- `main`: 가장 안정적인 기준 브랜치
- `apply`: 팀에서 통합 개발을 이어가는 기본 작업 브랜치
- `feat/<short-name>`: 큰 기능을 따로 개발할 때 생성
- `fix/<short-name>`: 버그 수정 전용 브랜치
- `backup/<short-name>`: 전환 전 임시 보존용 브랜치

## 운영 원칙

- 작은 수정은 `apply`에서 바로 작업하고, 검증 후 커밋/푸시합니다.
- 영향 범위가 크거나 실패 가능성이 높은 작업은 `feat/*` 브랜치에서 진행합니다.
- 브랜치 전환 전에는 반드시 현재 작업을 커밋하거나 `backup/*`으로 보존합니다.
- 다른 사람이 올린 작업은 바로 덮어쓰지 않고 먼저 fetch 후 비교합니다.
- 의미 없는 “update”, “fix” 같은 커밋 메시지는 피하고 결과 중심으로 작성합니다.

## 커밋 규칙

- 형식: 짧고 구체적으로 작성
- 예시:
  - `Add crowd API integration for trip search`
  - `Refine course recommendation UI on apply`
  - `Fix map rendering fallback for local demo`

## 푸시 규칙

- 검증 가능한 상태에서만 푸시합니다.
- 푸시 전에는 현재 브랜치와 변경 범위를 다시 확인합니다.
- `main`에는 직접 푸시하지 않고, 필요 시 `apply` 또는 기능 브랜치에서 정리 후 반영합니다.

## 머지 기준

- `apply`는 실제 개발 진행 브랜치로 유지합니다.
- 기능 브랜치는 작업이 끝나면 `apply`에 머지합니다.
- `main` 반영은 데모 안정화 시점에만 진행합니다.

## 현재 기준

- 이전 UI 실험 작업은 `backup/ui-wip-20260414`에 보존되어 있습니다.

## 담당 방식

- 브랜치 생성, 커밋, 푸시, 머지 전략은 Codex가 우선 관리합니다.
- 사용자가 별도 브랜치 전략을 원하면 그때 맞춰 조정합니다.
